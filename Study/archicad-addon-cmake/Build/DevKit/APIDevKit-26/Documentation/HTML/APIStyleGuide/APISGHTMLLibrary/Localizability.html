Localizability

All error messages and texts appearing to the user should come from 'STR#' resources or 'STRf' resources. The first type should contain all localizable texts; the second type can be used for non-localizable messages.

It is also important to separate debugging error messages ('Error 28' is probably not very meaningful to the user) from normal error messages which the user sees in the final version. The debugging option is usually indicated by the DEBUVERS compiler switch (you can also define that in the make environment when compiling the debug version). You should put all the debugging messages into a grc file in the RFIX folders, indicating that these strings don't need localization. You can also bracket these messages within #ifdef DEBUVERS - #endif, so that they don't go into the release version. (See the 'GALR' #32600 resource in the Attribute_ManagerFix.grc file.)

The same is true for other non-localizable strings (like names of library part parameters): they should also go into the files in the RFIX folder.

You should also be aware of multibyte systems (Japanese, Korean etc.); use the String Manager in the development kit to avoid such problems. The typical places where problems can occur are: static and edit text items, numeric input fields (e.g. degree symbol).

Commandments:

  • Create a correct interface: if you make mistakes (graphically incorrect, bad, incorrect, confusing wording, etc.), then these mistakes could go into 20 localized versions. These have to be corrected anyhow, and it makes a lot of difference in time and money if you have to do it in 20 versions.
  • Very good quality of the English text is a must: it is a crucial point in the usability of the software (and also for the localizers). Rely on the support of a language proofreader and interface designers. Never use slang or offending expressions (be aware of cultural differences!), and don't invent a new terminology -- the user will be confused.
  • Don't make unnecessary changes: don't modify the resources of a released add-on just for the sake of modification or minor advantages. These modifications have to be carried through 20 different language versions.
  • You are not ready when you finished coding: you have to add help and a documentation at minimum. This is the task of the programmer and the specificator.

Language problems:

  • Avoid concatenation: don't create sentences out of concatenated strings. Even if it worked in English, it will never work in any other language (order of words etc.). One sentence or expression should be one item/resource.
  • Using very short English expressions: expressions which can be expressed in English in a compact form could never be expressed in the same form in any other language. The interface should leave place for the longer localized strings (so that the localizer wouldn't have to rearrange the whole dialog).
  • Put comments behind the localizable resources: it is much easier for the localizer to figure out what that string means, or what its maximum length should be.
  • Never use the same string for different semantic meanings: this can be confusing for the localizer; besides it is very likely that the localized expression would be different in the different contexts in a foreign language.
  • Don't use abbreviations or play games with words.
  • Related information: should be stored side by side in the resource. If not, then place comments to indicate the relationship.
  • Changing menus and buttons: please place comments to indicate that a string from a resource would go into a menu or button.

Colors, icons:

  • Be careful in using colors and icons -- this is the job of the interface designer. Generally, you should use icons as replacements for text items (icons require less localization).
  • Don't put text into bitmaps, pictures or icons unless absolutely necessary.

Fonts:

  • Do not hardwire font names into the source code: put the font names into resources so that they would be localizable. Also leave place in the dialogs for bigger sized fonts (Mac vs Windows, Roman vs Japanese etc.). If you store font names, then leave place for longer font names (don't create checksums for 'Arial').

Delimiters, measurement units:

  • Don't hardwire decimal and thousand separators or measurement units into the source code: they all depend on a language/font/platform. Be prepared to handle double-byte versions as well (e.g. degree symbol).

About dialogs:

  • Don't use offsets: DG gives you the possibility of moving many dialog items at the same time with Offset. Use this only in the design phase; take it out from the release version.
  • DY: in a dialog use the same height for each item type. For example, LargePlain static/edit texts, radiobuttons should be 16, LargePlain checkboxes would be 18 pixel high. Use these values in all your dialogs! You can find the recommended values in the documentation of the Dialog Manager.
  • Overlapping/hidden items: avoid placing 20-30 dialog items in position (0, 0), then distributing them from the source code. Place them onto tab pages group by group at least, so that the localizers would see which items will appear together in the dialog.
  • Leave enough place for text items: replacing compact English terms with much longer German or Japanese terms is one of the most difficult tasks for the localizers. This hurts the original intent of the designer not only by making the dialog 'lighter', but also by forcing the localizer to make unnecessary abbreviations or to reorganize the dialog completely.
  • Give name and ID number to each resource: at least put some information about the usage of the resource into comments (after the resource type!). Please also indicate at least with initials the programmer of the dialog (so that the localizers can figure out who to contact).

General rules for writing resources:

  • Stick to a certain format: as shown in the API examples. For example for dialogs: put the item index at the beginning of the line between comments (like: "/* 23 */"), and put other comments at the end of the line -- some of the localization tools used internally by Graphisoft rely on this.
  • Don't limit the length of text items if possible: but if you have to, then indicate this with a comment (e.g. "/* max 32 chars */"), so that the localizer should also learn this. Imposing an uncountable limit (like 1024) is completely unnecessary...
  • Spring cleaning: do this once in a while. Unused dialogs, string resources should be deleted.

Make environment:

  • The make environment supplied to Localization should work on both platforms. Take into account the possibilities supplied by environment: use the common files (like EmuLocal.h for defining language-specific things). Don't hardwire into your source code the language of the add-on (e.g. don't use #define JAPANESE to distinguish in your source code between a Japanese and an English version). Localizers hate to touch the source code; they work with resources.
  • Avoid defining INT_APP in the resource file. The code page depends on the value of the language_version_APP define (unless you really want to see the international code page in your Korean add-on :)).
  • Create a correct 'VERS' resource: for example, update the copyright date on year change.
  • If possible avoid including header files from other directories (like #include "..\Src\Mydefs.h") in the grc file. The GRC compiler only supports one extra access path, and that is usually reserved for the RFIX\Images folder. You can adjust this by modifying the RESCONVFLAGS variable in the make environment.

Other:

  • Consider technology changes seriously: introducing a new technology may have a serious impact on localization as well.
  • Check whether the technology you use is available and localized in many system versions/platforms.
