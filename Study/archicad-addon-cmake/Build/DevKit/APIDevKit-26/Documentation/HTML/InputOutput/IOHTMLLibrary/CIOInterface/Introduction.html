Introduction into CIO interface


InputOutput module has two interfaces regarding the compatibility issues: the "traditional" one, where user code is linked against the InputOutput (import) library through decorated names (decorated method, function and class names), and the "binary" one, where functiontality of the InputOutput module is accessed through handmade virtual function tables residing in appropriate header files.

The "traditional" interface is fragile regarding the compatibility because decorated name based linking is compiler dependent (and compiler version dependent). Moreover subsequent versions of the InputOutput module may be (so surely will be) incompatible because name and parameter list of methods may change (for example due to addition of new default parameters) which causes changes in decorated names. As a result, a newer version of the InputOutput library (DLL) will cause run-time errors with the "old" program.

The "binary" compatible interface is a thin layer on the top of the InputOutput module which eliminates most of the cross-version and cross-compiler problems. It uses handmade virtual function tables (like in COM) to access the "internal" IO methods, which will remain compatible cross versions. The exposed virtual functions will never change to assure stable compatibility, only new functions will be added to virtual function tables (of course new classes and consequently new virtual function tables can be also safely added). Virtual function tables are handmade structures with stable and known layout to assure the cross-compiler compatibility.
