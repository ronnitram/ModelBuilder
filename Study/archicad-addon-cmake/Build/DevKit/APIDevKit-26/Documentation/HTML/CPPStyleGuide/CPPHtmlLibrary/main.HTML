C++ Style Guide

Introduction

This document contains the coding standard of Graphisoft, which sets the formal requirements for writing source code. It also describes a few rules and recommendations for the content.

General formatting rules

Naming conventions

Using underscore (_) at the beginning of names is forbidden.

#define names

Should be written in capitals; names consisting of more than one word should be separated by underscores (_). E.g.: PLATFORM, DEBUG_LEVEL_2, REFERENCE_CHECK.

Type names (struct, enum, typedef, class, namespace, template, constant names (const), enumerated values, function names

Use UpperCamelCase, capitalize the first letter of every word.

namespace UtilDefs {

typedef unsigned int UnsignedInt;
const size_t ConstantValue = 0;

void FooBarBaz ();

struct MyType;
enum DummyEnum {
    DummyValue1,
    DummyValue2
}

}

Variable and parameter names

Use lowerCamelCase: capitalize the first letter of every word, except the first one.

void Foo (ODB::Ref<OBB::Object> dummyRef)
{
    size_t variable;
    unsigned int debugLevelCheck;
}

Spacing

Indenting

The base unit of indenting is one tab, equal to 4 characters. You can use spaces for better alignment where needed.

Spaces

General rules for placing spaces (you can use more than one whitespace for better alignment):

  1. Leave one space before the first opening bracket ''character''.
  2. Leave one space after the comma ''character''.
  3. Leave one space before and after the equal sign ''character''.
  4. Leave one space before and after binary operators (e.g. *, +, !=, |).
  5. Don't put any space for unary operators (e.g. ++x, !x), pointers (e.g. x[y], *x), and member operators (e.g. a->b, a.b).
  6. Leave one space before and after the question mark and the colon characters of the ternary operator.
  7. Don't put spaces in every other case (e.g. ::, "str"_x, these belong to the "other operators" category).
a.b = sizeof (int);               // rule #5 ("."), #3 ("="), and #1 ("(")
double x, y, z;                   // rule #2 (",")
for (size_t i = 0; i != 10; ++i); // rule #1 ("("), #3 ("="), #4 ("!="), and #5 ("++i")
bool result = condition ? a : b;  // rule #6 ("?", ":")

Alignment

Align function names in function groups.

ODB::Ref<ODB::Object> DoThis ();
void                  DoThat ();

class Dummy {
    static void                                   Foo ();
    static const GS::Array<ODB::Ref<ODB::Object>> Bar ();

     explicit Dummy (int a);
              Dummy ();
};

Align the '=' equal sign and the values for multiple, related assignments.

const size_t StartIndex   = 0;
const size_t MiddleIndex  = 49;
const size_t EndIndex     = 99;
size_t       currentIndex = 0;

enum Token {
    Integer   = 25,
    Real      = 48,
    Character = 701
};

Other

Line length

The maximum length of lines is 120 characters.

Substatements in blocks

You should use the following rules when using control flow statements (e.g. for, if-else, etc.) requiring sub-statements:

  1. For one-liners you can choose between the block and non-block form.
  2. For embedded statements only the innermost sub-statement can have non-block form.
  3. For compound statements should be consistent: either all or none in block form.
for (UInt32 i = 0; i < 100; i++) // rule #1
    statement;

while (condition) {              // rule #1
    // body
}

if (condition1) {
    if (condition2)              // rule #2, and...
        statement1;
    else                         // ... rule #3
        statement2;
}

Compound control flow statements

Don't put a new line between the preceding and the following parts.

if (condition1) {
    // body
} else if (condition2) {
    // body
} else {
    // body
}

do {
    // body
} while (condition2);

try {
    // body
} catch (...) {
    // body
}

Expressions

  • When comparing with constants (e.g. i == 5) the constant should be the second operand.
  • Don't use C-style cast in C++ code.
  • Don't rely on automatic bool conversion; state the conditionexplicitly:
unsigned int i = f ();
if (i != 0) { // instead of: if (i)
    ...
}

char* p = g ();
if (p != nullptr) { // instead of: if (p)
    ...
}
  • Also, do not use the bool result of logical negation. E.g. use i == 0 instead of !i, and p == nullptr instead of !p. (Of course, bool variables and expressions can be negated, e.g. !b).
  • Logical expressions consisting of many parts should be placed on multiple lines. The opening curly brace should be on a separate line when you have multiple conditions:
if ((a > 2 && b < 1) ||
    (c > 10 && d == 0) ||
    (e == -5) ||
    (f < 5 && f > 1))
{
    // body
}
  • Put parentheses around complicated expressions and rarely used operators (e.g. where && and ||, or = and == both appear), so that readers don't have think about operator precedence.
bool a = (c == b);

Control flow statements

switch

switch (key) {
    case Value1:
        // body
        break;

    case Value2:
        // body
        // no break

    case Value3:
    case Value4:
        // body
        break;

    case Value5: {
        // body with local variables
        break;
    }

    default:
        // body
        break;
}

You must have a default branch (error handling!), and it should always be the last one. You should add a "no break" comment, if after the execution of a non-empty case branch execution will continue to the next case branch. You shouldn't add break after unconditional return or throw statements.

For short, compressed case branches (e.g. conversion) you can use the more readable compact switch form:

switch (code) {
    case 'I': return 1;
    case 'R': return 1;
    case 'B': return 1;
    case '\n': return 1;
    default: return 1;
}

for, while, do-while

Put a space between the '(', ';', or ')' characters if you leave any of the expressions out in the for statement header. E.g.: for ( ; ; ).

Don't use any assignments when you specify the condition.

try-catch-throw

Catch the exception as a constant reference if the exception is not of a simple type:

catch (const Type& exception)

Variable declarations and definitions

  • Declare and/or define all variables in the smallest possible scope.
  • Declare all variables where you can initialize them.
  • Tie the * for pointers, and the & for references to the type:
char* ptr = nullptr;
USize& index = oldIndex;
  • You should put all pointer and reference declarations into separate lines.
  • If a "variable" is really a constant, then say so: use the const qualifier:
const Vector& v = translation;
  • Do not use "naked" literal values (magic consts), assign them to const variables.
  • Naming conventions: see Variable and parameter names section.

Functions

Declarations

  • Leave a space between the parameter types and their names. You can move the function name to a new line if you have multiple modifier keywords:
static int cdecl
SetPointer (char* p);
  • Use the following indentation and alignment if a function has many parameters, or the declaration doesn't fit into one line:
String ConcatenateName (const String* part1,
                        const String* part2,
                        const String* part3,
                        const String* part4,
                        bool          appendNewLine = false);
  • Use the const qualifier for all logically constant parameters.
  • Pass bool parameter only in trivial cases, use enum-s otherwise.
  • Naming conventions: see Type names (struct, enum, typedef, class, namespace, template, constant names (const), enumerated values, function names section.

Implementation

  • Template:
void    PrintNumber (int n, short radix)
{
    // body
}
  • Write the function header according to the rules and recommendations set for function declarations.
  • Start the function's body with 1-tab indentation.
  • Leave two empty lines after the closing curly brace '}' of the function.
  • Avoid having too compact structures, using very complex expressions, or magical C tricks. Try to write short functions (SRP!).
  • Keep conditional expressions maximum 3 levels deep. Extract the rest to separate (possibly inline) functions.
  • Use asserts to check incoming parameters (preconditions) at the beginning of the function body.
  • See the Variable declarations and definitions section on local (and static) variable declarations.

Class definitions

Basic structure

Visibility

class Vector {
private:
    // private part

protected:
    // protected part

public:
    // public part
};

Put the opening curly bracket '{' onto a new line if the header of the class spans multiple lines (see inheritance). In some rare cases you can use more then one public, protected, or private sections. E.g. you may decide that you put those declarations (local types, e.g. enum-s) into the public (or protected) sections that are used in the private (or protected) sections. You can put those definitions at the beginning of the class:

class Vector {
public:
    typedef double Coord;
    // other public definitions also used by the private (protected) part

private:
    // private part

protected:
    // protected part

public:
    // public part
};

Template classes

template < class Type, int BufferSize = 16>
class Array {
    // body
};

You should treat template arguments the same way as function arguments.

Inheritance

class Derived: public Base1,
               public Base2,
               protected Base3,
               private Base4
{
    // body
};

You should always indicate the default private keyword. For virtual inheritance the order should be public virtual Base. The opening brace in case of single inheritance should go to the end of the header line (class ...).

Naming conventions

  • Class names: see Type names (struct, enum, typedef, class, namespace, template parameter section).
  • Template parameter names: seeType names (struct, enum, typedef, class, namespace, template parameter section).

Body

private section - typical layout, recommendations, rules

For better readability the default private keyword should always appear at the beginning of the class.

Put a // disabled comment after disabled methods (e.g. constructors). E.g.:

String (const String& source);              // disabled
String& operator= (const String& source);   // disabled
protected section - typical layout, recommendations, rules

Use protected data members sparingly; implement protected (or inline) methods instead.

public section - typical layout, recommendations, rules

Never use public data members.

Data member declarations
  • You should place every data member on a separate line, below each other.
  • Align multiple data members:
char* str;      // points to the allocated string (if not nullptr)
USize size;     // size of the string
  • Leave an empty line between logical groups of data members.
Method declarations
  • Put each parameter into a separate line if the method doesn't fit into one line, or has many parameters. Align the type and parameter names to each other. E.g.:
virtual Vector3D ComputePosition (const Matrix&  rot,
                                  Mode           mode,
                                  double         x,
                                  double         y,
                                  double         z) const;
  • Leave an empty line after the method for better readability in such cases.
  • Template method declaration style:
template <class U, class V> inline Pair& operator= (const Pair<U, V>& p);
  • Always align the constructor method names if you use the explicit or inline keyword. E.g.:
         String ();
explicit String (char c);
         String (const char* cStr);
inline   String (const char* charPtr, USize charCount);
explicit String (unsigned char* pStr);
         String (const String& source);
  • Always write the explicit keyword for single-parameter constructors (except the copy ctor) unless you want to use them for automatic conversion.
  • Always use const, except in very special cases.
  • Always write the virtual virtual keyword in front of virtual methods for better readability, even if that method has already inherited that property from its parent.

Class implementations

Methods

  • The order of methods in the class definition and implementation file should be the same.
  • Constructor style
Array::Array (USize internalCapacity, USize maxSize):
    base1 (initialCapacity),
    base2 (0, 25),
    member1 ("", 40),
    member2 (1),
{
    // body
}
  • Always align the ancestor classes' and the data member initializations, with the ancestor classes first, then the data members in the same order as they were declared in the class. You can also initialize the data members in the constructor body if that is more practical.
  • The same rules and recommendations apply to method header style as to the method declarations.
  • You should always use scoped references when accessing out-of-class names. E.g. refer to the global (OS specific) Sleep () function as ::Sleep ().
  • Follow the rules for function implementation in other method implementation cases.

namespace

namespace CLib {
    void* MemCopy (void* destPtr, void* srcPtr, UInt32 count);
    void* MemMove (void* destPtr, void* srcPtr, UInt32 count);
    void* MemSet (void* destPtr, int filler, UInt32 count);
}

You should indent with one tab the body of those namespace-s, which contain simple function declarations, type definitions (typedef), and constant values. If you define or implement a (bigger) class in a namespace, then indenting is not always practical; use this style instead:

namespace Math {

class Vector {
    // body
};

}

Always comment the last line for big namespace-s.

Except the main () function everything should go into namespace-s to avoid name collisions — especially when you use external libraries.

Do not use the using namespace X directive in header files.

Preprocessor directives

#if defined (XXX)

// body

#else

// body

#endif

Do not indent "normal" C++ code in conditional preprocessor statements; but indent other preprocessor statements with one tab. Put comments after directives which are longer than two screens, so that matching directives could easily be found.

Comments

  • You should write comments in English. Do not use accented characters.
  • You can find the recommended style in the Control/Template.hpp and Control/Template.cpp sample files.
  • You should enclose unused parameters in comments (/* ... */). Do not use the #pragma unused directive. If a release build would require that, use the UNUSED_PARAMETER macro instead.
  • You should always add comments:
    • For unusual solutions and contructs (omitting break-s, empty loop body, etc.).
    • If understanding the code would take too long for someone else.
    • If something is forbidden or not recommended for others.

File structure

  • Always use 4 spaces as tabs in all files.
  • The maximum length of the lines is 120 characters. Statements (function and method bodies) shouldn't even get close to this number!
  • You should put header guards into all header files. The name consists of the GS prefix, the namespace identifier, the file name, and the HPP suffix in capital letters. Separate the members with underscores ("_"). E.g.:
#if !defined GS_IO_LOCATION_HPP
#define GS_IO_LOCATION_HPP

// body

#endif

In this example the name of the header file is "Location.hpp", and the namespace is "IO". This is a special case of preprocessor directive: do not indent the lines within.

  • Write all the #include statements at the beginning of the source and header files, followed by the forward declarations (in headers). E.g.:
// -------------------------------- Includes ---------------------------------

#include "Base.hpp"

#include "Array.hpp"

#include "Stack.hpp"

// ---------------------------- Predeclarations ------------------------------

class SymbolSet;
  • The header file should stand on its own; there shouldn't be any need to #include any other file to compile it. In the .cpp file the first #include should be the matching .hpp header file; this ensures that the header file compiles on its own.
  • File names are case sensitive, their extension should be .cpp and .hpp, or .c and .h, depending on whether they could be used in purely C code. Every file name reference (#include, makefile) should be case sensitive.
  • Do not use the "." (parent) operator when you #include another header.
  • Preferably every class has its own header and source file, but if you have many closely related classes, put them into one file. A template class and its specialized versions should also be kept in one file. On the other hand, avoid placing a base class and its descendants into one file.
  • Every header file should #include only what the class declaration needs: the ancestor classes and the data member types. If a type is used only as a pointer, reference, or method return value in the class, then use forward declarations instead of #include to reduce the number of dependencies, and to speed up compilation.
  • The files of each subsystem (module, package) should be kept in a separate subdirectories.
