Test application for the Geometry module

GeometryTEST is an automatic testing tool for the Geometry module. Its testing facilities try to ensure the correct behavior of the functions in Geometry module.

  • GeometryTEST is a command line tool on Windows and an MPW tool on the MacOS, so you can run it from a batch or make script.
  • You can run GeometryTEST with randomly generated input, and it will verify if any error returns or exception occurs. In this case GeometryTEST will record the erroneous input into the test cases file to make error reproduction easy. It is a way to generate interesting test cases.
  • If you know tricky input cases you can enter them into the test case files, and try them. If GeometryTEST finds a test case where only the input is given, then - if no error occured with its input - will record the output of the function to the test cases file.
  • And last, if you know the input and the expected output, you can enter them and GeometryTEST will signal if the expected and produced output is different. If you change the algorithm of a function you can check the change - or uniformity - of the results with this feature.

Using GeometryTEST

GeometryTEST functionListFile [-a] [-rdd] [-p basepath] [-s | -t]

You can usually simply call the GeometryTEST with the function list file, and without any parameters.

The parameters in detail:

functionListFile
This simple text file contains the names of the functions to be tested. All entry is a row ended with a new line and all entry consists of a name and a path to the file with the test cases for that function.
-a
This switch causes the program to ignore the written test cases and making all tests with random parameters.
-rdd
If -a is set, the number after this switch determine the number of repeats of tests of each functions, else has no effect.
-p basepath
If you give here a base path, all relative paths given in the parameters and in the list files will be relative to this base path.
-s
Silent mode. GeometryTEST will show only a summary.
-t
Talkative mode. GeometryTEST will show all messages to help debugging.

Description of the functions list file

This file consists of two columns with whitespaces between then as separators.

  • the first column contains the name of the functions
  • the second one contains the name of the file which contain the test cases and optionally the results

After the ! character the program ignores the rest of the line so you can put some comments here.
The functions list file can include other functions list files with the #include command. It gives you the possibity to arrange the functions to be testing into different groups, and make collection files with #include-s for different test runs.


! Beginning

!
! This is a comment line
!

#include "SomeOtherFunction.txt" ! An include file


!__Column 1_________Column 2______

 FunctionName1 "FunctionDataFile1.txt" ! Here is possible comment, too.
 FunctionName2 "FunctionDataFile2.txt"
 

Description of the test cases file

Every tested function must have an own test case file. This file always begins with a header of the following three lines:

Best time = 1.0000000000e+300 ;
Worst time = 0.0000000000e-000 ;
Last time = 0.0000000000e-000 ;

It records performace information of the function, respectively the the best, worst, and last execution time.

After this header the test cases follow:

All test case must be followed by a semicolon. A test case is divided into two parts with a colon. The first part is the input, the second is the expected output.

Both the input and output parts consist of a list of parameters. The list elements must be separated with comma.

#1. example with static data members without rnd:
Best time = 1.0000000000e+300 ;
Worst time = 0.0000000000e-000 ;
Last time = 0.0000000000e-000 ;

!
! Circle_3Pt (<<Coord, <<Coord, <<Coord, >>Coord, >>bool);
!

(0.0, 0.0), (10.0, 0.0), (0.0, 10.0):

(none, none), true; ! fill the origo and check if is it OK (true)
